Cómo está construido

La arquitectura del sistema

👤

Huésped o interesado

Futuro punto de entrada: WhatsApp, Instagram, formulario web o bot conversacional.

🤖

IA / Canal de entrada (futuro)

Interpreta la consulta y la traduce a llamadas estructuradas. No toma decisiones críticas ni confirma reservas por sí sola.

⚙️

n8n — orquestador de workflows

Coordina el flujo operativo: consulta disponibilidad, crea pre-reservas, registra pagos y confirma reservas. Delega las reglas críticas a la base de datos.

🐘

Supabase / PostgreSQL — fuente de verdad

El backend real. Aquí viven las reglas de negocio, las funciones atómicas, los triggers de auditoría y la expiración automática de pre-reservas vía pg_cron.

👥

Vistas operativas y equipo humano

Franco y Rodrigo verifican pagos y aprueban confirmaciones. Las vistas SQL muestran el estado operativo en tiempo real.

El proceso completo

Flujo de una reserva

💬

Consulta

El huésped pregunta por fechas y cabaña por WhatsApp, web o bot.

📋

Registro

El sistema registra la consulta y verifica disponibilidad real.

🔒

Pre-reserva

Se bloquean las fechas temporalmente. El huésped tiene tiempo para reportar el pago.

💳

Pago reportado

El huésped reporta el pago. El sistema lo registra en estado en_revision y extiende el bloqueo de fechas.

🔍

Revisión manual

El equipo verifica la transferencia. El pago queda en revisión hasta su aprobación.

✅

Confirmación

Franco o Rodrigo verifican el pago y el sistema crea la RESERVA definitiva confirmada.
